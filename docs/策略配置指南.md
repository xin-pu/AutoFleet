# AutoFleet 策略配置指南

> 面向运维与开发：如何用标签和规则表达选机策略  
> 与 WinForms Demo（[`demo.md`](./demo.md)）中的写死数据一一对应  
> MES 处置边界见 [MES对接说明](./MES对接说明.md)

---

## 0. MES 与 AutoFleet 各管什么

| 谁 | 管什么 | 不管什么 |
|----|--------|----------|
| **MES** | pass/fail 后的 disposition、process_code、fail 次数、搬运 | 机台打分、标签权重 |
| **AutoFleet** | 规则匹配、候选过滤、标签打分、选机 | 合格判定、报废判定 |

正式 API 由 MES 传入 `disposition`；Demo 用**场景下拉**模拟 MES 已算好的 disposition + process_code。

---

## 1. 配置三层结构

```
基础数据（机台 + 标签绑定）
    ↓
调度规则（触发条件 + 打分项）
    ↓
运行时输入（MES: disposition + process_code；历程: fail_type 等）
```

**硬过滤**：同 `process_code`、在线、非维护。  
**软打分**：标签匹配（DCA、良率档、负载等）累加得分。

## 2. 标签体系（当前 Demo）

| 维度 | 取值 | 绑在谁身上 | 用途 |
|------|------|------------|------|
| DCA类型 | K0000、A0001 | 机台 | 与产品 DCA 一致才加分（`RequireProductInstrumentMatch`） |
| 良率档 | 高、低 | 机台 | Demo 静态模拟；正式版改为动态 snapshot |
| 负载 | 空闲 | 机台 | 首测偏好空闲机台 |
| 标记 | 刚测失败 | 动态 | 通信异常换机时贴在上次失败机台 |
| 区域 | Fail区 | Fail 区机台 | 报废路由 |

### Demo 机台与标签

| 机台 | 工序 | 标签 |
|------|------|------|
| M-A-01 | A | DCA=K0000, 良率档=高 |
| M-A-02 | A | DCA=K0000, 良率档=低, 负载=空闲 |
| M-A-03 | A | DCA=A0001, 良率档=高 |
| M-B-01 | B | DCA=A0001, 良率档=低 |
| M-B-02 | B | DCA=K0000, 良率档=高 |
| M-FAIL-01 | FAIL | 区域=Fail区 |

## 3. disposition → 下一工序

MES 在 **A 测完成后** 计算 disposition 与 `process_code`，再调调度 API：

| disposition | 下一工序 | Demo 场景 |
|-------------|----------|-----------|
| first_test | A | 首次进A |
| pass_to_b | B | A通过 → 去B |
| fail_retest | A | 项点不良复测 / 通信换机 |
| fail_scrap | FAIL | 去 Fail 区 |

> Demo 当前规则仍用 `last_test_result` + `fail_type=直接报废` 触发；正式版改为匹配 `disposition`，见 [MES对接说明 §7](./MES对接说明.md#7-规则匹配与-disposition-的关系)。

## 4. 规则清单（Demo 五条）

**目标形态**：路由规则匹配 `disposition`；`fail_type` 仅用于 `fail_retest` 下的选机细分。

**当前 Demo**（待 M3 重构为 disposition）：

| 优先级 | 规则名 | 触发概要 | 目标 disposition |
|--------|--------|----------|------------------|
| 5 | A失败-项点不良复测 | last fail + 项点不良 | fail_retest |
| 6 | A失败-直接报废去Fail区 | FAIL + 直接报废 | fail_scrap |
| 8 | A失败-通信异常换机 | last fail + 通信异常 | fail_retest |
| 10 | A首测/正常进站 | A + last 非 fail | first_test |
| 10 | A通过-流转B | B + last A pass | pass_to_b |

| 优先级 | 规则名 | 打分偏好 |
|--------|--------|----------|
| 5 | A失败-项点不良复测 | DCA匹配 + 高良率 +50，低良率 -30 |
| 6 | A失败-直接报废去Fail区 | Fail区 +100 |
| 8 | A失败-通信异常换机 | 避开刚测失败 -100，DCA匹配 |
| 10 | A首测/正常进站 | DCA匹配 + 低良率 + 空闲 |
| 10 | A通过-流转B | DCA匹配 + 略偏好高良率 |

**匹配原则**：按 priority **升序**，**第一条**触发满足即生效。

## 5. 六场景 walkthrough

### 场景 1：首次进A（SN-10086，K0000）

- 无测试历程 → 下一工序 **A**
- 命中：A首测/正常进站
- M-A-02 得：DCA匹配 +40，低良率 +15，空闲 +20 → **选 M-A-02**

### 场景 2：A通过去B（SN-30001，A0001）

- 上次 A pass → 下一工序 **B**
- 命中：A通过-流转B
- 仅 M-B-01 匹配 A0001 → **选 M-B-01**

### 场景 3：A失败复测（SN-20001，项点不良）

- 下一工序 **A**，命中项点不良复测规则
- M-A-01 高良率 +50 → **选 M-A-01**

### 场景 4：A失败换机（SN-40001，通信异常，上次 M-A-01）

- 动态标签「刚测失败」贴 M-A-01 → -100
- **选 M-A-02**

### 场景 5：A失败去Fail区（SN-50001，直接报废）

- 下一工序 **FAIL**（非跳过调度）
- **选 M-FAIL-01**

### 场景 6：A通过去B K0000（SN-60001）

- 下一工序 **B**
- **选 M-B-02**（K0000 匹配）

## 6. 打分项 match_type

| 值 | 含义 |
|----|------|
| 包含则加分 | 机台有该标签 → +分 |
| 不包含则加分 | 机台无该标签 → +分 |
| 包含则减分 | 机台有该标签 → -分 |
| 不包含则减分 | 机台无该标签 → -分 |

---

## 7. 新增场景怎么做

1. 标签管理：新维度/新值 → 绑到机台  
2. 新规则：写 trigger + 打分项 + priority  
3. 用 Demo 或 WPF 模拟调度验证  
4. 启用规则  

不必改调度引擎主流程。
